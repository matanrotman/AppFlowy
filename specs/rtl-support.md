# RTL/LTR Support

## Goal
Better right-to-left and left-to-right writing and display support ŌĆö both inside document content, and in the surrounding app UI (sidebar, toolbar) so the whole app feels natural in RTL languages like Hebrew or Arabic, not just the text you type.

## Current state
- The editor already has partial RTL handling ŌĆö a per-block text-direction attribute and a few targeted line-break fixes ŌĆö but full bidirectional support has been a known, long-requested gap since 2021 (AppFlowy issue #39: https://github.com/AppFlowy-IO/AppFlowy/issues/39). Mixed-direction content is the weakest part.
- The app's surrounding chrome (sidebar, toolbar, panels) has **no RTL awareness at all** today. The sidebar is pinned to the left edge using hardcoded pixel positioning (`Positioned(left: ...)` in `desktop_home_screen.dart`), not a direction-flexible layout ŌĆö so nothing in the app chrome currently follows interface language automatically.

## Desired behavior

### Phase 1 ŌĆö App chrome direction (sidebar + adjacent UI)
- A layout-direction setting that controls where the sidebar docks (left or right).
- **Auto-follow interface language by default**: switching the app's interface language to Hebrew/Arabic auto-flips the sidebar to the right; switching back to English (or another LTR language) flips it back to the left. A manual override is available in Settings for anyone who wants to pin a side regardless of language.
- Collapse/expand icon flips to match ŌĆö points toward the sidebar's current edge, not hardcoded left.
- Resize drag-handle logic is corrected so dragging toward the sidebar always widens it, regardless of which side it's docked on.
- Adjacent chrome sweep: toolbar/breadcrumb layout and any icons that visually imply "sidebar is on the left" (arrows, hover-reveal edge trigger) are checked and corrected as part of this phase, not left for later.

### Phase 2 ŌĆö Document content
- **Per-block direction: auto-detect, like Google Docs/Word.** A block's direction follows the first strong-direction character typed into it ŌĆö no manual per-block toggle needed. (Resolved open question ŌĆö see "To confirm with me" below.)
- **Mixed-direction lines: standard Unicode bidi.** A line's base direction follows the block (per above), but embedded runs of the other direction (an English word or a number inside a Hebrew sentence) stay in their own direction in place, matching standard browser/Word bidi behavior ŌĆö not whole-line-only direction. (Resolved open question.)
- **Block-insert ("+") menu opens the wrong side in RTL.** The hover controls next to each block (drag handle, "+" to add below, Ōīź-click to add above) are fine, but the block-insert menu that the "+" opens always expands to the right, even when the block/document is RTL ŌĆö it should open toward the natural reading direction, mirroring how sidebar popovers were fixed in Phase 1 (`sidebarPopoverDirection`/`documentPopoverDirection` helpers already exist as precedent).
- Correct cursor movement, text selection, and toolbar placement inside RTL blocks. **Resolved 2026-07-15** ŌĆö see Session Log for the four bugs below, all now fixed with headless test coverage.

### Related, but tracked separately
- **RTL table support** has its own spec: `specs/tables.md`. Its first phase (auto-aligning a table pasted from an RTL source) depends on nothing in this phase and can be built independently.
- **Drag-select icon margin** (the drag-handle/"+" icons sit close enough to the first word of a block to interfere with clicking it): not RTL-specific, applies in LTR too. **Fixed 2026-07-15** ŌĆö see Session Log.
- **Drag-selecting from a paragraph into/out of a table**: general selection-handling gap, not RTL-specific. Logged for a future session, not scoped into this spec or into `tables.md`.
- **Block-insert "+" on a non-empty block creates a new block below rather than opening the menu in place**: investigated 2026-07-15, turned out to be existing, by-design editor behavior (Notion-style "+"), not a bug ŌĆö not in scope for this spec.
- **A block created via Enter-split from an RTL sibling stays RTL even when pure English is then typed into it**: flagged 2026-07-14, not actively re-verified this session since it wasn't one of the items asked for ŌĆö but the direction-lookup fix from 2026-07-15 (`text_direction_mixin.dart` no longer depending on a mounted render state) very likely already fixes this as a side effect, and existing fork tests for "auto empty text don't use previous node direction because we can determine by the node text" already cover the same underlying logic. Worth a quick confirmation next time this comes up, not urgent.
- **Floating selection toolbar appears above the selection start instead of near the selection end** ŌĆö when the selection requires scrolling, this forces scrolling back up to reach the toolbar. **Fixed 2026-07-15** ŌĆö turned out to be app-side and not RTL-specific; see Session Log.

- Existing but narrowly-scoped: `LayoutDirection`/`AppFlowyTextDirection` in `appearance_cubit.dart`, currently only wraps the document editor (`plugins/document/presentation/editor_page.dart`) in `Directionality` ŌĆö Phase 2 will extend this; Phase 1 needs a separate, new app-chrome-level setting since this one doesn't touch the sidebar.

- `frontend/appflowy_flutter/lib/plugins/document/application/document_appearance_cubit.dart` ŌĆö default text direction fallback (now `"auto"` instead of unset).
- `frontend/appflowy_flutter/lib/workspace/application/settings/appearance/appearance_cubit.dart` ŌĆö `AppFlowyTextDirection.fromTextDirectionPB`'s default case (now `auto` instead of `ltr`).
- `frontend/appflowy_flutter/lib/plugins/document/presentation/editor_plugins/slash_menu/slash_command.dart` ŌĆö passes the current block's direction into the (forked) `SelectionMenu`.
- `frontend/appflowy_flutter/lib/plugins/document/presentation/editor_configuration.dart` ŌĆö `actionTrailingBuilder`, hover-icon-to-text spacing.

## Out of scope (for now)
- Mobile layout direction (this plan targets desktop first, matching how the sidebar/menu system is currently desktop-specific).
- Localizing/translating the UI itself into Hebrew/Arabic strings (separate effort from layout direction).
