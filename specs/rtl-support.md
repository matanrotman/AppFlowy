# RTL/LTR Support

## Goal
Better right-to-left and left-to-right writing and display support — both inside document content, and in the surrounding app UI (sidebar, toolbar) so the whole app feels natural in RTL languages like Hebrew or Arabic, not just the text you type.

## Current state
- The editor already has partial RTL handling — a per-block text-direction attribute and a few targeted line-break fixes — but full bidirectional support has been a known, long-requested gap since 2021 (AppFlowy issue #39: https://github.com/AppFlowy-IO/AppFlowy/issues/39). Mixed-direction content is the weakest part.
- The app's surrounding chrome (sidebar, toolbar, panels) has **no RTL awareness at all** today. The sidebar is pinned to the left edge using hardcoded pixel positioning (`Positioned(left: ...)` in `desktop_home_screen.dart`), not a direction-flexible layout — so nothing in the app chrome currently follows interface language automatically.

## Desired behavior

### Phase 1 — App chrome direction (sidebar + adjacent UI)
- A layout-direction setting that controls where the sidebar docks (left or right).
- **Auto-follow interface language by default**: switching the app's interface language to Hebrew/Arabic auto-flips the sidebar to the right; switching back to English (or another LTR language) flips it back to the left. A manual override is available in Settings for anyone who wants to pin a side regardless of language.
- Collapse/expand icon flips to match — points toward the sidebar's current edge, not hardcoded left.
- Resize drag-handle logic is corrected so dragging toward the sidebar always widens it, regardless of which side it's docked on.
- Adjacent chrome sweep: toolbar/breadcrumb layout and any icons that visually imply "sidebar is on the left" (arrows, hover-reveal edge trigger) are checked and corrected as part of this phase, not left for later.

### Phase 2 — Document content
- **Per-block direction: auto-detect, like Google Docs/Word.** A block's direction follows the first strong-direction character typed into it — no manual per-block toggle needed. (Resolved open question — see "To confirm with me" below.)
- **Mixed-direction lines: standard Unicode bidi.** A line's base direction follows the block (per above), but embedded runs of the other direction (an English word or a number inside a Hebrew sentence) stay in their own direction in place, matching standard browser/Word bidi behavior — not whole-line-only direction. (Resolved open question.)
- **Block-insert ("+") menu opens the wrong side in RTL.** The hover controls next to each block (drag handle, "+" to add below, ⌥-click to add above) are fine, but the block-insert menu that the "+" opens always expands to the right, even when the block/document is RTL — it should open toward the natural reading direction, mirroring how sidebar popovers were fixed in Phase 1 (`sidebarPopoverDirection`/`documentPopoverDirection` helpers already exist as precedent).
- Correct cursor movement, text selection, and toolbar placement inside RTL blocks. **Partially resolved as of 2026-07-14** — the floating-toolbar stale-position bug is fixed and confirmed via a widget test that pumps the real widget through a scroll; the icon-margin gap is bumped to 30px; the multi-run mid-character cursor bug is still genuinely open (see "Known related bugs" below). All three, plus the empty-RTL-line cursor case, still want a live glance before being called fully done.

### Related, but tracked separately
- **RTL table support** has its own spec: `specs/tables.md`. Its first phase (auto-aligning a table pasted from an RTL source) depends on nothing in this phase and can be built independently.
- **Drag-select icon margin** (the drag-handle/"+" icons sit close enough to the first word of a block to interfere with clicking it): not RTL-specific, applies in LTR too. **Bumped to 30px on 2026-07-14** — mechanism re-confirmed via test (LTR and RTL), pending a live glance to confirm 30px itself looks right. See "Known related bugs" below.
- **Drag-selecting from a paragraph into/out of a table**: general selection-handling gap, not RTL-specific. Logged for a future session, not scoped into this spec or into `tables.md`.
- **Block-insert "+" on a non-empty block creates a new block below rather than opening the menu in place**: investigated 2026-07-15, turned out to be existing, by-design editor behavior (Notion-style "+"), not a bug — not in scope for this spec.
- **A block created via Enter-split from an RTL sibling stays RTL even when pure English is then typed into it**: flagged 2026-07-14, not actively re-verified this session since it wasn't one of the items asked for — but the direction-lookup fix from 2026-07-15 (`text_direction_mixin.dart` no longer depending on a mounted render state) very likely already fixes this as a side effect, and existing fork tests for "auto empty text don't use previous node direction because we can determine by the node text" already cover the same underlying logic. Worth a quick confirmation next time this comes up, not urgent.

### Known related bugs, not yet scoped
Status as of 2026-07-14 (see Session Log entry same date for full detail):
- **Cursor renders mid-character/mid-token in real (multi-run) mixed Hebrew/English text — still genuinely open.** Investigated headlessly this session with a widget test built from the actual repro sentence (not a toy case): the "mid-token inside a date" claim is confirmed and now has a passing regression test; a narrower, distinct anomaly (the caret jumping to the wrong side right after a trailing comma exits an embedded run) is also confirmed but deliberately left un-fixed and documented via a `skip`ped test — a fix attempt (per-character `getBoxesForSelection` lookup) produced byte-identical output to the existing approach, so the issue sits inside Flutter's own bidi run classification, not this wrapper. Needs a reference implementation to compare against or a live look to define "correct" before attempting a real fix. `appflowy_rich_text.dart` (fork), `getCursorRectInPosition`; `caret_bidi_test.dart`.
- **Cursor far from the actual typing position on a fresh empty RTL line — re-run as a sanity check, still wants a live look.** The existing headless empty-line tests (RTL paragraph, RTL heading, LTR paragraph) all pass; no code changed. Not independently re-confirmed live yet.
- **Floating selection toolbar lands at a stale position after a scroll-triggered selection — fixed 2026-07-14, pending live confirmation.** Root cause was a timing bug, not a math bug: `DesktopFloatingToolbar` read render-object geometry synchronously in `initState()` (which runs during a frame's build phase, before that frame's layout), so a widget recreated in the same frame a scroll offset changes read stale, pre-scroll geometry. Fixed by deferring the read to a post-frame callback. Confirmed via a new widget test that pumps the real widget through an actual scroll and reproduces the stale position pre-fix, passes post-fix. `desktop_floating_toolbar.dart`; `desktop_floating_toolbar_test.dart`.
- **Icon-margin gap — bumped to 30px 2026-07-14, pending live confirmation** the value itself looks right. `editor_configuration.dart`.

- Existing but narrowly-scoped: `LayoutDirection`/`AppFlowyTextDirection` in `appearance_cubit.dart`, currently only wraps the document editor (`plugins/document/presentation/editor_page.dart`) in `Directionality` — Phase 2 will extend this; Phase 1 needs a separate, new app-chrome-level setting since this one doesn't touch the sidebar.
- Editor package (https://github.com/AppFlowy-IO/appflowy-editor) — forked to `https://github.com/matanrotman/appflowy-editor` (branch `rtl-direction-aware-selection-menu`) for the block-insert menu direction fix; `pubspec.yaml`'s `appflowy_editor` dependency points at the fork. - `frontend/appflowy_flutter/lib/plugins/document/application/document_appearance_cubit.dart` — default text direction fallback (now `"auto"` instead of unset).
- `frontend/appflowy_flutter/lib/workspace/application/settings/appearance/appearance_cubit.dart` — `AppFlowyTextDirection.fromTextDirectionPB`'s default case (now `auto` instead of `ltr`).
- `frontend/appflowy_flutter/lib/plugins/document/presentation/editor_plugins/slash_menu/slash_command.dart` — passes the current block's direction into the (forked) `SelectionMenu`.
- `frontend/appflowy_flutter/lib/plugins/document/presentation/editor_configuration.dart` — `actionTrailingBuilder`, hover-icon-to-text spacing.

## Out of scope (for now)
- Mobile layout direction (this plan targets desktop first, matching how the sidebar/menu system is currently desktop-specific).
- Localizing/translating the UI itself into Hebrew/Arabic strings (separate effort from layout direction).
