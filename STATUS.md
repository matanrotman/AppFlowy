- **Document RTL Phase 2 — status as of 2026-07-14:**
  - Auto-detect block direction (first-strong-character, like Google Docs) and mixed-direction Unicode bidi — implemented, verified working. Solid.
  - Block-insert ("+"/slash) menu direction — implemented via the editor fork (`03719b8a`), with a headless test confirming LTR/RTL genuinely resolve differently. Not flagged as broken by the user in the last live round. A separate, unrelated finding — clicking "+" on an existing non-empty block creates a new block below rather than opening the menu in place — turned out to be existing, by-design editor behavior (Notion-style "+"), not a bug.
  - **Icon-margin gap — bumped 12px → 30px this session**, per the user's explicit instruction. Strengthened test confirms the production value (not an arbitrary test value) renders as a real, undiminished gap in both LTR and RTL. Live glance still pending.
  - **Floating selection toolbar — fixed this session, live glance pending.** Root cause was a Flutter timing bug, precisely diagnosed via code reading: `DesktopFloatingToolbar.initState()` read render geometry synchronously during a frame's build phase, before that frame's layout phase had caught the scrolled content up to its new position — so a widget recreated in the same frame a scroll happens (exactly what the outer toolbar's scroll-debounce does) read stale, pre-scroll geometry. Fixed with a post-frame-callback deferral. Confirmed via a new widget test that pumps the *real* widget (through a real root `OverlayEntry`) through an actual scroll-then-remount sequence — reproduces a 350+ px stale-position miss against the old code, passes against the fix.
  - **Cursor renders mid-character inside a date embedded in a long mixed sentence — still genuinely open, not force-closed.** Investigated deeply this session using a widget test built from the literal repro sentence (not a toy case). Confirmed the reported "mid-token" claim directly — now a passing regression test. Beyond that, diagnosed the underlying mechanism precisely: most apparent "jumps" at bidi run boundaries are actually consistent with an already-validated design rule (caret sits adjacent to the preceding run), but one narrower, specific case (right after a trailing comma exits an embedded run) resolves inconsistently with that same rule. A fix attempt (per-character box lookup instead of the global affinity flag) produced byte-identical output to the existing code — the inconsistency sits inside Flutter's own bidi run classification, not in anything the app/fork wrapper controls. Left as an honestly-`skip`ped test rather than forcing a fix without real confidence, or claiming "fixed" without genuine verification — exactly the trap flagged after the 2026-07-15 correction. Needs either a reference implementation to compare against or the user's live judgment on what "correct" should mean here.
  - **Cursor far from typing position on an empty RTL line — re-confirmed headlessly, live glance still pending.** Existing tests (RTL paragraph, RTL heading, LTR paragraph) all still pass; no code changed.
  - **This session's verification approach**: per the user's explicit instruction not to drive their screen, all four items were investigated/fixed using headless `flutter test` widget tests built from real repro content (the actual failing sentence, a real scroll-then-remount sequence, the actual production gap value) — not synthetic toy cases, and with tests that pump the *real* widgets, not just extracted helper functions in isolation. ## Local build quick-reference

```
cargo make code_generation                                            # locale keys + freezed (only needed after clean/pubspec changes)
cd appflowy_flutter && flutter run -d macos                           # launch the app
```
